Leave `Possible Intended Reference` blank if uncertain.

Do not use catalog existence, control similarity, topic similarity or another valid control elsewhere in the document to infer the intended reference.

Use only this `Action to Take` value in the separate malformed-reference report:

`Manually review malformed reference`

The allowed final action-register `Action to Take` values in section 11 do not apply to the separate malformed-reference report.

## 9. Existing referenced control quality and relevance check

For every explicit valid `ISM-####` reference found in an eligible numbered `Doco` solution section, do not assume that the referenced control is a good match merely because it appears in the document.

Each explicitly referenced control must be assessed against the actual non-comment content of the section in which it appears.

This is a quality and relevance check of controls already referenced in `Doco`.

The purpose is to identify whether the document’s Existing Control reference agrees with the implementation evidence in that section, and, where it does not, to suggest a better fitting control that is present as an `ISM-control` object in the selected ASD OSCAL ISM catalog.

Do not remove, hide or replace explicitly referenced controls. Always preserve explicitly referenced controls from eligible solution sections in the output for traceability.

For each explicitly referenced control, assign one of the following internal `Referenced Control Relevance Assessment` values:

* `Agrees with section evidence`;
* `Partially agrees with section evidence`;
* `Does not agree with section evidence`;
* `Insufficient evidence to assess`;
* `Not present in selected ASD ISM catalog`.

Use these meanings:

* `Agrees with section evidence` means the section evidence directly supports the substantive requirement of the exact ISM control statement, without relying on broad topic matching or assumptions.
* `Partially agrees with section evidence` means the referenced control is related to the same sentence, bullet, table row, paragraph or immediately adjacent control-evidence block, but that same evidence only partly supports the substantive requirement of the referenced control or more directly supports another control that is present as an `ISM-control` object in the selected ASD OSCAL ISM catalog. Do not mark an existing control as partially agreeing merely because different evidence elsewhere in the same section supports an additional control.
* `Does not agree with section evidence` means the referenced control does not have a direct, defensible, statement-level relationship to the same sentence, bullet, table row, paragraph or immediately adjacent control-evidence block in which the reference appears.
* `Insufficient evidence to assess` means the section explicitly references the control but does not contain enough implementation-specific evidence to confirm whether the control is appropriate.
* `Not present in selected ASD ISM catalog` means the explicit reference was found in `Doco`, but the normalised control ID is not present in the selected ASD OSCAL ISM catalog after mandatory validation.

Where an explicit ISM reference appears in a dedicated control-mapping, compliance-mapping or traceability table, treat the same table row as the relevant control-evidence block. Use only the evidence in that row and any directly referenced eligible numbered section. Do not use other rows in the table as evidence for that control.

For every `Existing control already in Doco` row, assign one of the following `Existing Control Finding` values:

* `No issue identified`;
* `Minor relevance concern`;
* `Potentially incorrect reference`;
* `Insufficient evidence`;
* `Catalog validation issue`.

Map the values as follows:

| Referenced Control Relevance Assessment | Existing Control Finding        | Action to Take                     |
| --------------------------------------- | ------------------------------- | ---------------------------------- |
| Agrees with section evidence            | No issue identified             | Keep existing control              |
| Partially agrees with section evidence  | Minor relevance concern         | Review existing control            |
| Does not agree with section evidence    | Potentially incorrect reference | Review existing control            |
| Insufficient evidence to assess         | Insufficient evidence           | Review existing control            |
| Not present in selected ASD ISM catalog | Catalog validation issue        | Catalog validation review required |

When assessing agreement, compare the section evidence against the exact OSCAL control statement first.

Do not assess agreement using only:

* ISM topic names;
* ISM guideline names;
* broad keywords;
* general cyber security relevance;
* assumptions about what the design probably intended;
* prior knowledge of ISM control families;
* replacement, successor or related controls.

Do not mark a referenced control as wrong merely because another control might also apply.

Only mark a referenced control as `Does not agree with section evidence` where the mismatch is clear from the section content and the exact ISM control statement.

When uncertain, prefer `Insufficient evidence to assess` over `Does not agree with section evidence`.

## 12. Suggested alternate controls for existing referenced controls

For every explicit ISM reference in an eligible solution section, perform an active better-fit and supplement check.

A `Suggested alternate control` may be produced where the same sentence, bullet, table row, paragraph or immediately adjacent control-evidence block more directly or more precisely supports a different ISM control than the one explicitly referenced in `Doco`.

Do not limit alternate checks only to existing controls assessed as partially aligned or incorrect. If the existing control is defensible but a different control is clearly a more precise fit for the same evidence block, mark the existing control as `Minor relevance concern` and add one `Suggested alternate control`.

Where an explicitly referenced control is assessed as:

* `Partially agrees with section evidence`;
* `Does not agree with section evidence`; or
* `Insufficient evidence to assess`;

actively check whether the same evidence block clearly explains what the original reference was trying to cover and whether a different catalog control is a better fit.

Use this decision rule:

`Same evidence block and better fit = Suggested alternate control.`

`Different implementation detail = Suggested control.`

`Same evidence block and complementary control, not better fit = Suggested control.`

Treat evidence as the same evidence only where the existing ISM reference appears in the same sentence, bullet, table row, paragraph or immediately adjacent control-evidence block as the implementation detail being reviewed.

Where an explicit ISM reference appears in a dedicated control-mapping, compliance-mapping or traceability table, treat the same table row as the relevant control-evidence block. Use only the evidence in that row and any directly referenced eligible numbered section. Do not use other rows in the table as evidence for that control.

Treat evidence as separate evidence where it addresses a different implementation detail, component, process, setting, network path, account type, security activity or system behaviour.

If the evidence is separate from the explicit reference, classify any action-register suggestion as a `Suggested control`, not a `Suggested alternate control`.

Suggest an alternate control only if the section contains implementation-specific evidence that has a defensible control-statement-level relationship to a different control that is present as an `ISM-control` object in the selected ASD OSCAL ISM catalog and that more directly or more precisely represents the same implementation evidence or intended control objective as the existing explicit reference.

The alternate control must satisfy the action-register matching gates in section 14.

Do not suggest an alternate merely because:

* the existing control looks unusual;
* the section is broadly related to another ISM topic;
* another control is more common;
* another control would be good practice;
* another control is a replacement, successor, superseded or related control;
* the alternate requires assumptions not stated in the section.

If a better fitting alternate exists, add a separate row for the same eligible numbered `Doco` section with:

* `Output Category` = `Suggested alternate control`;
* `Existing Control` = the normalised explicit `ISM-####` control that the alternate is reviewing;
* `Existing Control Finding` = the finding for the existing control being reviewed;
* `Suggested Alternate Control` = the suggested alternate ISM control;
* `Suggested New Control` = blank;
* `Action to Take` = `Replace or supplement existing control with suggested alternate`.

The `Rationale` column must explain both:

1. why the alternate control substantively supports or more precisely matches the section evidence; and
2. why it is a better fit than the explicitly referenced control named in `Existing Control`.

Use this wording:

`Suggested as an alternate because the section explicitly states <evidence>, which substantively supports the suggested ISM control statement requiring <control intent>. The existing referenced control <ISM-####> received finding <Existing Control Finding> because <reason>.`

Do not add more than one suggested alternate control for the same explicitly referenced control.

Do not add more than two suggested alternate controls for the same eligible numbered section.

A suggested alternate control does not change the document’s existing control reference. It is a quality-review finding only.

## 13. Suggested controls — balanced evidence-led matches

For each `Eligible solution section`, actively consider whether the section contains implementation-specific evidence that supports one or more ISM controls.

Do not default to `No Suggested control` merely because the section does not quote or closely mirror the ISM control wording.

Add a `Suggested control` to the ISM Action Register where:

* the section contains specific implementation evidence; and
* the candidate control is present in the selected catalog; and
* the evidence substantively supports the control statement or control intent; and
* the rationale can clearly explain the relationship between the section evidence and the ISM control.

The evidence does not need to satisfy every element of the control statement. However, it must be more than a broad product name, framework reference, capability label or generic cyber security topic.

Use `Suggested control` where the suggested control is based on separate implementation evidence in the same eligible section and does not directly correct, replace or better represent an existing explicit reference.

Use this decision rule before classifying any suggested control:

`Same evidence block and better fit = Suggested alternate control.`

`Different implementation detail = Suggested control.`

`Same evidence block and complementary control, not better fit = Suggested control.`

Do not classify a control as a suggested alternate merely because it appears in the same section as an existing reference.

If a candidate control is based on different implementation evidence from the explicit control reference, classify it as `Suggested control` even if an existing reference in the same section is partially aligned, misaligned or insufficiently supported.

Treat evidence as different implementation evidence where it addresses a different component, configuration item, control setting, network path, account type, log source, backup item, patching activity, vulnerability process, security tool, data flow or operational activity from the existing explicit reference.

Do not suggest a control merely because the section is about the same general topic as the ISM topic, subsection or guideline.

The ISM control statement must be the primary basis for the match. ISM topic, subsection and guideline titles may be used only as supporting context, not as the reason for selection.

Use the section’s specific evidence phrases, technologies, configurations, security activities, risks or requirements.

For each eligible solution section, build a candidate pool of likely controls using exact phrases, key nouns, technologies, configurations, activities, data flows and control-statement terms from the local indexed catalog.

Then classify candidates as:

* `Suggested control` where the match is direct, substantive and defensible enough for the action register;
* `Optional Review Candidate` where the match is plausible and section-specific but not strong enough for the action register;
* omitted where the match is generic, keyword-only, assumption-based or not section-specific.

Match against the indexed ISM control statement first. Use topic, subsection and guideline context only to disambiguate between otherwise plausible controls.

For suggested controls, first build a small candidate pool using exact phrase, key noun, technology, activity, configuration and control-statement searches against the local in-memory indexed catalog only. Do not use web search, external search results or sources outside the selected catalog.

Final inclusion must satisfy the action-register matching gates in section 14.

Do not match on broad keywords alone.

Reduce candidates to the strongest direct or substantive statement-level matches.

For `Suggested control` rows:

* `Output Category` = `Suggested control`;
* `Existing Control` = blank;
* `Existing Control Finding` = blank;
* `Suggested Alternate Control` = blank;
* `Suggested New Control` = the new suggested ISM control;
* `Action to Take` = `Add new control`.

## 13A. Optional review candidates

If `Optional review candidates = Off`, do not output Optional Review Candidates anywhere.

If `Output mode = Full coverage strict`, do not output Optional Review Candidates anywhere, even if `Optional review candidates = On`.

Optional Review Candidates are non-authoritative human-review items that sit outside the ISM Action Register. They are intended to surface plausible, section-specific ISM controls for human review where the evidence is implementation-specific but does not meet the final action-register confidence threshold.

They must never appear in the ISM Action Register.

Include an Optional Review Candidate where:

* the section is an Eligible solution section;
* the section contains specific implementation evidence, configuration, mechanism, data flow, access model, logging model, hardening setting, backup or resilience design, vulnerability activity, or operational security process;
* the candidate control is present in the selected catalog; and
* the candidate has a defensible, control-statement-level relationship to the section evidence, even if the match is not strong enough for the ISM Action Register.

Do not include generic keyword-only, framework-only, product-name-only, good-practice-only, assumption-based, replacement, successor, related-control or unrelated controls.

Where an eligible solution section contains implementation-specific security evidence but no final action-register suggested control, actively consider whether one or two Optional Review Candidates should be produced.

Do not include more than one Optional Review Candidate for the same eligible section unless the section contains two clearly separate implementation-specific evidence items. Do not include more than two Optional Review Candidates for any eligible section.

Optional Review Candidate rows do not count as action-register rows. If an eligible section has only optional review candidates and no existing, suggested alternate or suggested control row in the final action register, still add a `No action-register ISM control identified` row unless Output mode is `Mapped controls only`.

If `Optional review candidates = On` and optional candidates are produced, place them outside the final ISM Action Register in a clearly separated worksheet or section named:

`Optional Review Candidates`

Use these columns:

| Doco Numbered Section / Subsection | Section Title / Topic | Candidate ISM Control | Candidate ISM Control Description | Trigger Phrase | Gate Not Met | Reason for Human Review |
| ---------------------------------- | --------------------- | --------------------- | --------------------------------- | -------------- | ------------ | ----------------------- |

Use `Gate Not Met` to identify why the candidate was not elevated into the ISM Action Register, such as `Evidence not specific enough for action-register inclusion`, `Evidence incomplete for action-register inclusion`, or `Directness below action-register threshold`.

Do not include optional candidates if they are merely broad, weak, generic, keyword-only, framework-only, product-name-only, assumption-based, good-practice, replacement, successor, related-control or speculative items.

If `Optional review candidates = On` but no optional candidates are produced, omit the Optional Review Candidates worksheet or section.

## 14. Action-register evidence-led matching gates

Before including any `Suggested alternate control` or `Suggested control` in the ISM Action Register, apply the following gates.

A control may be included in the ISM Action Register where the gates are substantially satisfied.

Do not require exact wording alignment. The section evidence must directly or substantively support the ISM control statement or control intent, and the rationale must be explainable without relying on broad assumptions.

The evidence does not need to restate every word of the ISM control, but it must describe a specific implementation, configuration, mechanism, process, data flow, security setting, or operational activity that substantively satisfies, supports or is governed by the control statement.

If the section contains specific implementation evidence but one gate is not fully met, consider whether the control should be included as an Optional Review Candidate under section 13A.

### Gate 1 — Eligible section gate

The section is classified as an `Eligible solution section`.

Do not suggest controls for excluded non-solution sections.

### Gate 2 — Implementation evidence gate

The `Doco` section states an actual or proposed configuration, mechanism, process, setting, rule, data flow, system behaviour or operational activity.

Generic mentions of technologies, frameworks, tools, standards, compliance objectives or capability names are insufficient for action-register inclusion unless the same section also describes the specific implementation detail that makes the control applicable.

### Gate 3 — Statement or control-intent alignment gate

The implementation evidence directly or substantively aligns with the substantive requirement or control intent of the exact ISM control statement, not merely the control topic, subsection or guideline area.

The match can be included where the section evidence substantively satisfies, supports, enforces, implements or is governed by the ISM control statement, even if the section does not restate every element of the control wording.

If the ISM control statement cannot be connected to the section evidence without relying on broad topic association or unstated assumptions, do not include the control in the ISM Action Register. Consider Optional Review Candidates if section 13A permits it.

### Gate 4 — Specificity gate

The evidence identifies the relevant component, system, account type, network path, log source, protocol, cryptographic setting, backup item, patching process, vulnerability process, security tool or operational activity.

### Gate 5 — No-assumption gate

The match does not require unsupported assumptions about architecture, maturity, effectiveness, governance, ownership, operational practice, scope, inheritance, assurance or controls implemented elsewhere.

Reasonable interpretation of explicit section evidence is allowed. Do not require the document to spell out operational details that are not necessary to establish the control relationship.

### Gate 6 — Best-control gate

The control is one of the strongest available statement-level matches for that exact section, not one of several broadly related controls.

Where two or three controls each match separate implementation details in the same section, each may be included subject to section 15.

### Gate 7 — Lowest-level section gate

The control is mapped to the most specific numbered section containing the evidence, not to a parent/container section unless the parent itself contains the actual evidence.

Do not copy controls from parent sections to child sections.

Do not copy controls from child sections to parent sections.

### Gate 8 — Evidence explanation gate

The reason for applicability can be explained using specific wording or evidence from the `Doco` section.

The reason or review finding must not rely on generic phrases such as:

* `broadly relates to`;
* `supports` without explaining how;
* `is relevant to`;
* `may help with`;
* `could be considered`;
* `is good practice for`;
* `is generally associated with`;
* `is implied by`;
* `would normally be expected`;
* `appears to align`;
* `is likely applicable`.

If a gate is clearly not satisfied, do not include the control in the ISM Action Register.

When uncertain, prefer an Optional Review Candidate over a weak action-register mapping where section 13A permits it.

Suggested-control reasoning must use language such as:

`Included because the section explicitly states <evidence>, which substantively supports the ISM control statement requiring <control intent>.`

For `Suggested alternate control` rows, reasoning must also explain why the alternate is a better fit than the existing referenced control.

Do **not** include a suggested control in the ISM Action Register if it is only:

* generally related to cyber security;
* related only through a broad keyword such as `logging`, `access`, `network`, `patching`, `monitoring`, `backup`, `identity`, `firewall`, `encryption`, `MFA`, `RBAC`, `SIEM`, `hardening`, `vulnerability`, `incident`, `proxy`, `TLS` or `Key Vault`;
* an upstream governance, policy, assurance or risk-management control not directly addressed by the section;
* a control that would require unsupported assumptions about implementation maturity, compliance status, effectiveness or operating procedure;
* a control that applies only to another system component or a different `Doco` section;
* merely desirable but not clearly grounded in the section content;
* a replacement, related, successor or superseded control not explicitly present as an `ISM-control` object in the selected ASD OSCAL ISM catalog.

When uncertain about an existing reference, prefer `Insufficient evidence to assess` over suggesting a weak alternate.

## 15. Suggested-control limits

For each `Eligible solution section`:

* Prefer one strong Suggested control where the section contains clear implementation-specific security evidence.
* Include a second or third Suggested control where the section contains separate, distinct implementation details that map to different ISM control statements.
* Do not include more than three Suggested controls for any section.
* Do not include more than two suggested alternate controls for any section.
* Do not use these limits as targets.
* Do not add suggested controls to parent/container sections when the same evidence is mapped to a child section.
* Do not add suggested controls to short sections containing only headings, architecture labels, generic descriptions, lists of components or transitional wording.
* Do not add suggested controls for generic references to `ISM alignment`, `PROTECTED`, `Essential Eight`, `security tooling`, `monitoring`, `hardening`, `backup`, `patching`, `MFA`, `RBAC`, `logging`, `SIEM`, `SOAR`, `SOC`, `vulnerability management`, `application control`, `encryption` or `network security` unless the section states the specific implementation detail that directly or substantively supports the ISM control statement or control intent.

A section describing a security capability does not automatically justify mapping every ISM control related to that capability.

For example:

* a section about logging does not automatically justify all logging controls;
* a section about TLS does not automatically justify all cryptography controls;
* a section about privileged access does not automatically justify all access-management controls;
* a section about patching does not automatically justify all vulnerability-management controls;
* a section about backups does not automatically justify all resilience or recovery controls;
* a section about network restrictions does not automatically justify all gateway, firewall or segmentation controls;
* a section about MFA does not automatically justify all authentication controls;
* a section about RBAC does not automatically justify all authorisation, administration or privileged access controls;
* a section about PROTECTED alignment does not automatically justify classification, cryptography, gateway, hardening or governance controls.

### Column rules

#### Doco Numbered Section / Subsection

Use the eligible numbered section from `Doco`.

Use document order, not lexical order.

#### Section Title / Topic

Use the section heading from `Doco`.

If the heading is unclear, provide a concise topic summary based only on the section content.

#### Output Category

Use only:

* `Existing control already in Doco`;
* `Suggested alternate control`;
* `Suggested control`;
* `No action-register ISM control identified`.

#### Trigger Phrase

Populate this for every row.

Use one concise evidence phrase, configuration detail, table value or setting that triggered the row.

#### Existing Control

Populate this only where the row relates to a valid explicit ISM control already referenced in an eligible solution section of `Doco`.

For `Suggested alternate control` rows, populate this with the existing control being reviewed.

Leave blank for `Suggested control` rows.

Leave blank for `No action-register ISM control identified` rows.

#### Existing Control Finding

Populate this for `Existing control already in Doco` rows.

For `Suggested alternate control` rows, repeat the finding for the existing control being reviewed.

Use only:

* `No issue identified`;
* `Minor relevance concern`;
* `Potentially incorrect reference`;
* `Insufficient evidence`;
* `Catalog validation issue`.

Leave blank for `Suggested control` and `No action-register ISM control identified` rows.

#### Suggested Alternate Control

Populate this only for `Suggested alternate control` rows.

Use the normalised `ISM-####` value.

Leave blank for all other row types.

#### Suggested New Control

Populate this only for `Suggested control` rows.

Use the normalised `ISM-####` value.

Leave blank for all other row types.

#### Action to Take

Use only the allowed action-register values.

Do not create custom action text.

#### Existing Control Description

Use the description-column population rules in section 18.

#### Suggested Control Description

Use the description-column population rules in section 18.

#### Mapped / Suggested ISM Guideline Section, Mapped / Suggested ISM Guideline Subsection and Mapped / Suggested ISM Topic

Populate from the selected ASD OSCAL ISM catalog for the control being actioned in the row.

For `Existing control already in Doco` rows, these hierarchy fields apply to the existing referenced control.

For `Suggested alternate control` rows, these hierarchy fields apply to the suggested alternate control, not the existing referenced control.

For `Suggested control` rows, these hierarchy fields apply to the Suggested control.

For controls not present in the selected catalog, leave these fields blank.

Do not invent or infer missing hierarchy values.

When uncertain, use the nearest available catalog ancestor titles and leave ambiguous higher-level fields blank rather than guessing.

#### Rationale

Use concise rationale.

For existing controls with finding `No issue identified`, use:

`Keep existing control because the section explicitly states <evidence>, which directly matches the ISM control statement requiring <control intent>.`

For existing controls with finding `Minor relevance concern`, use:

`Review existing control because the section explicitly states <evidence>, which only partially matches the ISM control statement.`

For existing controls with finding `Potentially incorrect reference`, use:

`Review existing control because the section explicitly states <evidence>, which does not directly match the Existing Control statement.`

For existing controls with finding `Insufficient evidence`, use:

`Review existing control because it is explicitly referenced in Doco, but the section does not provide enough implementation-specific evidence to confirm direct applicability.`

For existing controls with finding `Catalog validation issue`, use:

`Catalog validation review required because the control is explicitly referenced in Doco but is not present in the selected ASD OSCAL ISM catalog after mandatory validation.`

For suggested alternates, use:

`Suggested as an alternate because the section explicitly states <evidence>, which substantively supports the suggested ISM control statement requiring <control intent>. The existing referenced control <ISM-####> received finding <Existing Control Finding> because <reason>.`

For Suggested controls, use:

`Add new control because the section explicitly states <evidence>, which substantively supports the ISM control statement requiring <control intent>.`

For `No action-register ISM control identified` rows, use:

`No action-register ISM control identified because the section does not contain enough implementation-specific evidence to support an action-register ISM control.`
