## 10. Existing referenced control quality and relevance check

For every explicit valid `ISM-####` reference found in an eligible numbered `Doco` solution section, do not assume that the referenced control is a good match merely because it appears in the document.

Each explicitly referenced control must be assessed against the actual non-comment content of the section in which it appears.

This is a quality and relevance check of controls already referenced in `Doco`.

The purpose is to identify whether the document’s existing ISM control reference agrees with the implementation evidence in that section, and, where it does not, to suggest a better fitting control that is present as an `ISM-control` object in the selected ASD OSCAL ISM catalog.

Do not remove, hide or replace explicitly referenced controls. Always preserve explicitly referenced controls from eligible solution sections in the output for traceability.

For each explicitly referenced control, assign one of the following internal `Referenced Control Relevance Assessment` values:

* `Agrees with section evidence`;
* `Partially agrees with section evidence`;
* `Does not agree with section evidence`;
* `Insufficient evidence to assess`;
* `Not present in selected ASD ISM catalog`.

Use these meanings:

* `Agrees with section evidence` means the section evidence directly supports the substantive requirement of the exact ISM control statement, without relying on broad topic matching or assumptions.
* `Partially agrees with section evidence` means the referenced control is related to the section content, but the section evidence more directly supports another control that is present as an `ISM-control` object in the selected ASD OSCAL ISM catalog, or only partly supports the substantive requirement of the referenced control.
* `Does not agree with section evidence` means the referenced control does not have a direct, defensible, statement-level relationship to the section content.
* `Insufficient evidence to assess` means the section explicitly references the control but does not contain enough implementation-specific evidence to confirm whether the control is appropriate.
* `Not present in selected ASD ISM catalog` means the explicit reference was found in `Doco`, but the normalised control ID is not present in the selected ASD OSCAL ISM catalog after mandatory validation.

For every `Existing control already in Doco` row, assign one of the following `Existing Control Finding` values:

* `No issue identified`;
* `Minor relevance concern`;
* `Potentially incorrect reference`;
* `Insufficient evidence`;
* `Catalog validation issue`.

Map the values as follows:

| Referenced Control Relevance Assessment | Is Existing Control Appropriate?        | Existing Control Finding        | Action to Take                     |
| --------------------------------------- | --------------------------------------- | ------------------------------- | ---------------------------------- |
| Agrees with section evidence            | Yes                                     | No issue identified             | Keep existing control              |
| Partially agrees with section evidence  | Partially                               | Minor relevance concern         | Review existing control            |
| Does not agree with section evidence    | No                                      | Potentially incorrect reference | Review existing control            |
| Insufficient evidence to assess         | Insufficient evidence                   | Insufficient evidence           | Review existing control            |
| Not present in selected ASD ISM catalog | Not present in selected ASD ISM catalog | Catalog validation issue        | Catalog validation review required |

When assessing agreement, compare the section evidence against the exact OSCAL control statement first.

Do not assess agreement using only:

* ISM topic names;
* ISM guideline names;
* broad keywords;
* general cyber security relevance;
* assumptions about what the design probably intended;
* prior knowledge of ISM control families;
* replacement, successor or related controls.

Do not mark a referenced control as wrong merely because another control might also apply.

Only mark a referenced control as `Does not agree with section evidence` where the mismatch is clear from the section content and the exact ISM control statement.

When uncertain, prefer `Insufficient evidence to assess` over `Does not agree with section evidence`.

## 12. Output category values and action model

Use only the following `Output Category` values in the final output:

* `Existing control already in Doco`;
* `Suggested alternate control`;
* `Suggested additional control`;
* `No clear ISM control identified`.

Use only the following `Action to Take` values in the final action register:

* `Keep existing control`;
* `Review existing control`;
* `Replace or supplement existing control with suggested alternate`;
* `Add new control`;
* `No action — no clear ISM control identified`;
* `Catalog validation review required`.

The allowed final action-register `Action to Take` values apply only to the consolidated ISM control action register. The separate malformed-reference report may use only `Manually review malformed reference`.

Apply the action values as follows:

| Output Category                  | Condition                                                              | Action to Take                                                  |
| -------------------------------- | ---------------------------------------------------------------------- | --------------------------------------------------------------- |
| Existing control already in Doco | Existing referenced control agrees with section evidence               | Keep existing control                                           |
| Existing control already in Doco | Existing referenced control partially agrees with section evidence     | Review existing control                                         |
| Existing control already in Doco | Existing referenced control does not agree with section evidence       | Review existing control                                         |
| Existing control already in Doco | Existing referenced control has insufficient evidence to assess        | Review existing control                                         |
| Existing control already in Doco | Existing referenced control is not present in selected ASD ISM catalog | Catalog validation review required                              |
| Suggested alternate control      | Better fitting control suggested for an existing referenced control    | Replace or supplement existing control with suggested alternate |
| Suggested additional control     | New control suggested from separate strong section evidence            | Add new control                                                 |
| No clear ISM control identified  | Eligible solution section has no existing or suggested control         | No action — no clear ISM control identified                     |

Do not use free-text action values in the final action register.

Use `Review existing control` for the existing-control row, and use `Replace or supplement existing control with suggested alternate` only on the separate `Suggested alternate control` row where a very-high-confidence alternate is actually identified.

Do not treat a suggested alternate control as an automatic replacement. It is a quality-review recommendation. The final action wording must allow the document owner to decide whether to replace, supplement or retain the original reference.

## 13. Suggested alternate controls for existing referenced controls

Where an explicitly referenced control is assessed as either:

* `Partially agrees with section evidence`; or
* `Does not agree with section evidence`;

perform a very-high-confidence alternate-control check.

Where an explicitly referenced control is assessed as:

* `Insufficient evidence to assess`;

create a `Suggested alternate control` only if the same section contains direct implementation evidence that clearly explains what the original reference was trying to cover.

Use this mini decision rule before classifying any suggested control:

`Same sentence/table row/control-evidence block = alternate candidate; different implementation detail = additional candidate.`

Treat evidence as the same evidence only where the existing ISM reference appears in the same sentence, bullet, table row, paragraph or immediately adjacent control-evidence block as the implementation detail being reviewed.

Treat evidence as separate evidence where it addresses a different implementation detail, component, process, setting, network path, account type, security activity or system behaviour.

If the evidence is separate from the explicit reference, classify any very-high-confidence suggestion as a `Suggested additional control`, not a `Suggested alternate control`.

Suggest an alternate control only if the section contains direct, implementation-specific evidence that clearly aligns to a different control that is present as an `ISM-control` object in the selected ASD OSCAL ISM catalog and that better represents the same implementation evidence or intended control objective as the existing explicit reference.

The alternate control must pass all very-high-confidence matching gates.

Do not suggest an alternate merely because:

* the existing control looks unusual;
* the section is broadly related to another ISM topic;
* another control is more common;
* another control would be good practice;
* another control is a replacement, successor, superseded or related control;
* the alternate requires assumptions not stated in the section.

If a better fitting alternate exists, add a separate row for the same eligible numbered `Doco` section with:

* `Output Category` = `Suggested alternate control`;
* `Existing ISM Control Already in Doco` = the normalised explicit `ISM-####` control that the alternate is reviewing;
* `Is Existing Control Appropriate?` = the appropriateness value of the existing control being reviewed;
* `Existing Control Finding` = the finding for the existing control being reviewed;
* `Suggested Suitable Alternate Control` = the suggested alternate ISM control;
* `New Suitable ISM Control to Add` = blank;
* `Action to Take` = `Replace or supplement existing control with suggested alternate`.

The `Rationale` column must explain both:

1. why the alternate control directly matches the section evidence; and
2. why it is a better fit than the explicitly referenced control named in `Existing ISM Control Already in Doco`.

Use this wording:

`Suggested as an alternate because the section explicitly states <evidence>, which directly matches the suggested ISM control statement requiring <control intent>. The existing referenced control <ISM-####> was assessed as <assessment> because <reason>.`

Do not add more than one suggested alternate control for the same explicitly referenced control.

Do not add more than two suggested alternate controls for the same eligible numbered section.

A suggested alternate control does not change the document’s existing control reference. It is a quality-review finding only.

## 14. Suggested additional controls — very high confidence only

For each `Eligible solution section`, the default outcome is:

`No suggested additional control`.

Only add a `Suggested additional control` where the section contains direct, implementation-specific evidence that strongly aligns to the exact ISM control statement.

Use `Suggested additional control` where the suggested control is based on separate implementation evidence in the same eligible section and does not directly correct, replace or better represent an existing explicit reference.

Use this mini decision rule before classifying any suggested control:

`Same sentence/table row/control-evidence block = alternate candidate; different implementation detail = additional candidate.`

Do not classify a control as a suggested alternate merely because it appears in the same section as an existing reference.

If a candidate control is based on different implementation evidence from the explicit control reference, classify it as `Suggested additional control` even if an existing reference in the same section is partially aligned, misaligned or insufficiently supported.

Treat evidence as different implementation evidence where it addresses a different component, configuration item, control setting, network path, account type, log source, backup item, patching activity, vulnerability process, security tool, data flow or operational activity from the existing explicit reference.

## 15. Very-high-confidence matching gates

Before including any `Suggested alternate control` or `Suggested additional control`, apply all of the following gates.

A control may be included only if **all** gates pass.

### Gate 1 — Eligible section gate

The section is classified as an `Eligible solution section`.

Do not suggest controls for excluded non-solution sections.

### Gate 2 — Implementation evidence gate

The `Doco` section states an actual or proposed configuration, mechanism, process, setting, rule, data flow, system behaviour or operational activity.

Generic mentions of technologies, frameworks, tools, standards, compliance objectives or capability names are insufficient.

### Gate 3 — Statement alignment gate

The implementation evidence directly aligns with the substantive requirement of the exact ISM control statement, not merely the control topic, subsection or guideline area.

If the ISM control statement cannot be quoted or paraphrased into a direct requirement that clearly matches the section evidence, do not suggest the control.

### Gate 4 — Specificity gate

The evidence identifies the relevant component, system, account type, network path, log source, protocol, cryptographic setting, backup item, patching process, vulnerability process, security tool or operational activity.

### Gate 5 — No-assumption gate

The match does not require assumptions about architecture, maturity, effectiveness, governance, ownership, operational practice, scope, inheritance, assurance or controls implemented elsewhere.

### Gate 6 — Best-control gate

The control is one of the strongest available statement-level matches for that exact section, not one of several broadly related controls.

### Gate 7 — Lowest-level section gate

The control is mapped to the most specific numbered section containing the evidence, not to a parent/container section unless the parent itself contains the actual evidence.

Do not copy controls from parent sections to child sections.

Do not copy controls from child sections to parent sections.

### Gate 8 — Evidence explanation gate

The reason for applicability can be explained using specific wording or evidence from the `Doco` section.

The reason or review finding must not rely on generic phrases such as:

* `broadly relates to`;
* `supports`;
* `is relevant to`;
* `may help with`;
* `could be considered`;
* `is good practice for`;
* `is generally associated with`;
* `is implied by`;
* `would normally be expected`;
* `appears to align`;
* `is likely applicable`.

If any gate fails, do not suggest the control.

When uncertain, do not include the control.

Suggested-control reasoning must use language such as:

`Included because the section explicitly states <evidence>, which directly matches the ISM control statement requiring <control intent>.`

For `Suggested alternate control` rows, reasoning must also explain why the alternate is a better fit than the existing referenced control.

Do **not** include a suggested control if it is only:

* generally related to cyber security;
* related only through a broad keyword such as `logging`, `access`, `network`, `patching`, `monitoring`, `backup`, `identity`, `firewall`, `encryption`, `MFA`, `RBAC`, `SIEM`, `hardening`, `vulnerability`, `incident`, `proxy`, `TLS` or `Key Vault`;
* an upstream governance, policy, assurance or risk-management control not directly addressed by the section;
* a control that would require assumptions about implementation maturity, compliance status, effectiveness or operating procedure;
* a control that applies only to another system component or a different `Doco` section;
* merely desirable but not clearly grounded in the section content;
* a replacement, related, successor or superseded control not explicitly present as an `ISM-control` object in the selected ASD OSCAL ISM catalog.

When uncertain, prefer `No clear ISM control identified` over a weak mapping for sections without explicit references.

When uncertain about an existing reference, prefer `Insufficient evidence to assess` over suggesting a weak alternate.

## 16. Suggested-control limits

For each `Eligible solution section`:

* Prefer zero suggested controls.
* Include one suggested additional control only where the evidence is direct and implementation-specific.
* Include a second suggested additional control only where the section contains a separate, distinct implementation detail that maps directly to a different ISM control statement.
* Do not include more than two suggested additional controls for any section.
* Do not include more than two suggested alternate controls for any section.
* Do not use the two-control limits as targets.
* Do not add suggested controls to parent/container sections when the same evidence is mapped to a child section.
* Do not add suggested controls to short sections containing only headings, architecture labels, generic descriptions, lists of components or transitional wording.
* Do not add suggested controls for generic references to `ISM alignment`, `PROTECTED`, `Essential Eight`, `security tooling`, `monitoring`, `hardening`, `backup`, `patching`, `MFA`, `RBAC`, `logging`, `SIEM`, `SOAR`, `SOC`, `vulnerability management`, `application control`, `encryption` or `network security` unless the section states the specific implementation detail that directly matches the ISM control statement.

A section describing a security capability does not automatically justify mapping every ISM control related to that capability.

For example:

* a section about logging does not automatically justify all logging controls;
* a section about TLS does not automatically justify all cryptography controls;
* a section about privileged access does not automatically justify all access-management controls;
* a section about patching does not automatically justify all vulnerability-management controls;
* a section about backups does not automatically justify all resilience or recovery controls;
* a section about network restrictions does not automatically justify all gateway, firewall or segmentation controls;
* a section about MFA does not automatically justify all authentication controls;
* a section about RBAC does not automatically justify all authorisation, administration or privileged access controls;
* a section about PROTECTED alignment does not automatically justify classification, cryptography, gateway, hardening or governance controls.
