### v8 repeatability hardening controls

The following controls are mandatory and override looser wording elsewhere in the prompt:

1. **Stable-ledger prerequisite for mapping repeatability** â€” suggested controls, suggested alternate controls and optional review candidates are repeatable only when generated from the same supplied frozen `Section Eligibility Ledger` and `Evidence Cluster Ledger`. Do not use two independently rebuilt ledgers as proof of mapping repeatability.
2. **No inferred renumbering** â€” never create, renumber, shift or repair section numbers because an unnumbered heading, file-tree heading, appendix label, figure caption, table caption or styled paragraph appears between numbered sections. If the section number cannot be obtained deterministically from an accepted heading source, do not create a numbered section for that paragraph.
3. **Ledger ID schema is mandatory** â€” `Ledger Row ID` must match `^SEL-[0-9]{4}$`; `Candidate Row ID` must match `^CAL-[0-9]{5}$`; `Register Row ID` must match `^IAR-[0-9]{5}$`; `Evidence Cluster ID` must match `^EC-SEL-[0-9]{4}-[PTBSF]-[0-9]{4}-[0-9]{4}-[0-9a-f]{8}$`; `Canonical Section Key` must begin with `SELKEY-`. Any different prefix such as `LED-`, `ECL-`, freeform IDs or sequential evidence IDs is a validation failure.
4. **Evidence clusters are first-class artefacts** â€” the `Evidence Cluster Ledger` worksheet is the authoritative cluster handoff. The `Evidence Cluster Inventory` field in the `Section Eligibility Ledger` is a compact summary only and must not be the sole cluster source where Excel output is supported.
5. **Mechanical eligibility formula** â€” eligibility must be derived from locked boolean fields, not a narrative judgement alone. A row is eligible only when no hard exclusion applies and all three evidence-triad fields are `Yes`.
6. **Candidate-ledger completeness** â€” in `Full run` or `Mapping from supplied frozen ledger`, where `Eligible solution sections > 0`, a zero-row `Candidate Assessment Ledger` is a validation failure unless the run is `Section ledger only` or an explicit hard-stop occurred before mapping.
7. **Rejected candidates are audit evidence** â€” every assessed candidate, including rejected candidates, must be recorded. A `No ISM control identified` row is not permitted unless rejected candidates or a locked no-candidate support record prove that candidate discovery was attempted for that section.
8. **Suggested alternates are anchored and transparent** â€” every `Suggested alternate control` must identify the specific existing explicit control row it reviews and must use the same `Ledger Row ID` and `Evidence Cluster ID`. Every weak existing-control row must also show whether named alternate candidates were considered, how many were considered, the best rejected alternate where applicable, and the exact reason no alternate was accepted.
9. **No bare alternate rejection for weak existing controls** â€” for existing controls with `Minor relevance concern`, `Potentially incorrect reference` or `Insufficient evidence`, the rejection reason `No stronger same-evidence alternate identified` is insufficient unless the Candidate Assessment Ledger also records the alternate discovery method, alternate-candidate count, best rejected alternate control ID where at least one alternate was considered, and the statement-level failure reason.
10. **Optional candidates are uncertainty overflow only** â€” Evidence Score `2` may appear only in `Optional Review Candidates` and must record the action-register gate that was not met. It must never be promoted to the ISM Action Register to improve coverage.
11. **Optional-candidate cap transparency** â€” when Optional Review Candidates are enabled, report both the number of candidates before the global cap and the number omitted by the cap. Do not hide cap suppression.
12. **Validation metadata is byte-stable** â€” catalog published metadata and frozen-ledger/mapping-mode validation wording must be normalised to the locked display values in section 20 so two repeatable mapping runs do not differ only because of workbook date serials, equivalent phrasing, source-filename carry-over, or baseline-context wording.
13. **Frozen-ledger metadata is separated from current-run extraction metadata** â€” in `Mapping from supplied frozen ledger` mode, the current run did not extract `Doco`. Therefore, `Doco extraction performed this run` must be `No`, `Doco body extraction hash` must be `Not applicable â€” Frozen ledger input`, and any carried extraction hash from the supplied ledger must be recorded only in `Frozen source Doco body extraction hash`.
14. **Mapping-equivalence hash controls workbook-level drift** â€” calculate `Mapping equivalence hash` over the stable mapping artefact sheets only, excluding `Validation Summary`, `Repeatability Audit`, workbook properties, timestamps, styles and other run-context metadata. Use this hash to distinguish true mapping drift from expected baseline-assessment metadata differences.

If any v8.1 hardening control cannot be satisfied, produce the workbook with failed validation where permitted, or stop with the applicable hard failure message where the prompt requires a hard stop.

A section with explicit ISM references but no substantive solution-design or implementation evidence is not automatically eligible. It is eligible only if the section also contains implementation-specific solution evidence or is a control-mapping, compliance-mapping or traceability row that contains direct implementation evidence for the referenced control.

A section may be eligible where it identifies a named technology, component, service, security tool, integration, data flow, access model, monitoring path, operational process or control responsibility and describes its role in the proposed or implemented solution. Do not require low-level configuration values where the section still provides enough implementation-specific evidence to identify what security function is being performed and where it applies.

A numbered section is an `Eligible solution section` only where its assigned body text contains all three evidence-triad elements:

1. a named component, service, process, configuration item, data flow, account type, system, security tool or operational activity;
2. a concrete implementation action, configuration, restriction, workflow, rule, integration, operating practice or control responsibility; and
3. a security outcome, protection mechanism, monitoring behaviour, assurance requirement, access-control purpose, resilience outcome or risk-treatment purpose.

Headings, table titles, table headers, figure captions, architecture labels, product names, component inventory, capability names and broad framework terms alone are not enough.

A numbered section is an `Eligible solution section` only if it also contains substantive solution-design content describing a specific implemented or proposed:

* security configuration;
* technical component;
* data flow;
* access control;
* authentication or authorisation mechanism;
* network path or firewall rule;
* cryptographic setting;
* logging, monitoring or alerting configuration;
* backup, resilience or recovery configuration;
* vulnerability, patching or hardening activity;
* security tool deployment or integration;
* operational security process directly tied to the solution design.

A numbered section must be classified as `Excluded non-solution section` if it is any of the following:

* front matter;
* document control, approvals, reviewers, distribution or classification content;
* notices;
* table of contents;
* table of figures;
* table of tables;
* introductory, background or purpose-only content;
* document map or architecture map-only content;
* in-scope or out-of-scope descriptions that only describe project scope and not solution implementation;
* intended audience;
* terms, definitions, acronyms or glossary content;
* figure-only or diagram-only content with no explanatory solution text;
* table-only content that lists document structure rather than solution configuration;
* parent/container section with no substantive solution content of its own;
* section that only states that another document, policy, SSP, CCM, HLD, LLD or external artefact will address the control;
* section that references frameworks, standards or compliance objectives without describing a solution implementation;
* section that only lists products, tools, headings or capability names without describing their solution role, implementation function, integration, data flow, configuration, operational use or control responsibility.

If eligibility is genuinely ambiguous after applying the rules above, prefer `Excluded non-solution section` unless the section itself contains at least one implementation-specific evidence phrase that identifies both a solution component/process and its security function.

### Mandatory mechanical eligibility fields and formula

For every `Section Eligibility Ledger` row, populate these locked boolean fields before assigning `Eligibility Classification`:

* `Hard Exclusion Triggered` = `Yes` where any hard exclusion in the decision matrix applies; otherwise `No`.
* `Hard Exclusion Reason` = the first matching hard-exclusion rule label, or blank where no hard exclusion applies.
* `Named Component Present` = `Yes` only where the assigned body text names a concrete component, service, process, configuration item, data flow, account type, system, security tool, role or operational activity.
* `Implementation Action Present` = `Yes` only where the assigned body text states a concrete action, configuration, restriction, workflow, rule, integration, operating practice, monitoring activity, backup activity, patching activity, hardening activity or control responsibility.
* `Security Outcome Present` = `Yes` only where the assigned body text states a security outcome, protection mechanism, monitoring behaviour, assurance requirement, access-control purpose, resilience outcome, integrity outcome, confidentiality outcome or risk-treatment purpose.
* `Explicit ISM Reference Present` = `Yes` only where at least one valid explicit `ISM-####` reference is found in the section.
* `File-tree / Code-layout Only` = `Yes` where the assigned body text is only file paths, route paths, folder trees, Ansible tree layout, package names, module names, generated code structure, command listings or artefact lists without an enforceable security behaviour.
* `Figure-caption Only` = `Yes` where the assigned body text is only a figure caption, diagram label, architecture label or table/figure cross-reference.
* `Eligibility Formula Result` = the mechanical result below.

Use this exact formula:

`Eligibility Formula Result = Eligible` only when `Hard Exclusion Triggered = No` AND `Named Component Present = Yes` AND `Implementation Action Present = Yes` AND `Security Outcome Present = Yes`.

Otherwise set `Eligibility Formula Result = Excluded`.

Set `Eligibility Classification = Eligible solution section` only when `Eligibility Formula Result = Eligible`. Set `Eligibility Classification = Excluded non-solution section` when `Eligibility Formula Result = Excluded`.

The `Eligibility Reason` must cite the boolean fields that drove the result, not only a narrative judgement. Example: `Excluded â€” Hard Exclusion Triggered = Yes; File-tree / Code-layout Only = Yes` or `Eligible â€” component, implementation action and security outcome all present; no hard exclusion triggered`.

### Deterministic eligibility decision matrix

Apply the following eligibility decision matrix in this exact order. Stop at the first matching rule.

1. Always classify sections `1`, `1.x`, document control, purpose, scope, in-scope, out-of-scope, intended audience, terms, definitions, document map, table of contents, table of figures, table of tables, notices, approvals, distribution, classification, glossary and equivalent front-matter or introductory content as `Excluded non-solution section`. This is a hard exclusion even where valid explicit ISM references appear. Valid explicit ISM references in these sections are counted in excluded-section metrics only and must not create action-register rows, suggested controls or optional review candidates.

2. Classify figure-only, diagram-only or caption-only sections as `Excluded non-solution section`. A figure caption, table caption, diagram title or cross-reference is not body text for eligibility unless the section also contains explanatory prose or table content that independently satisfies the eligible-section rule.

3. Classify file-tree, folder-layout, code-layout, Ansible tree, repository tree, route-list, placeholder, appendix-list or artefact-list sections as `Excluded non-solution section` unless the same section states a specific security control obligation applied to the listed items. If the only assigned evidence after a heading consists of path lists, Ansible tree layout, file names, folder names, repository tree content, route paths, generated code layout, commands or artefact lists, classify the section as `Excluded non-solution section` even if the heading is security-related or appears near eligible security sections.

4. Classify deployment-logistics-only sections as `Excluded non-solution section` where they only describe rollout sequencing, manual versus automated deployment, pilot/global rollout, migration staging, inventory attachment, or as-built documentation, and do not state a security control behaviour or configuration.

5. Classify table-only sections as `Eligible solution section` only where the table contains concrete configuration values or enforceable security rules. Eligible table examples include firewall rules, access roles and permissions, audit rules, log categories, retention periods, cryptographic settings, authentication requirements, allow/block rules, backup schedules, vulnerability scan settings, patch timeframes, hardening settings or monitoring rules.

6. Classify table-only sections as `Excluded non-solution section` where the table only lists component inventory, VM sizing, product names, document structure, file paths, captions, generic roles, placeholders or examples without an enforceable security behaviour.

7. Classify policy, register or record sections as `Eligible solution section` only where the section identifies the governed subject and required fields, approval/review obligation, retention obligation, monitoring obligation or enforcement mechanism. Otherwise classify as `Excluded non-solution section`.

8. If a section contains explicit ISM references but fails the eligibility rules above, classify it as `Excluded non-solution section`; record the explicit references in the ledger for excluded-reference counting only.

9. If eligibility remains ambiguous after rules 1 to 8, classify as `Excluded non-solution section`.


After classification, assign deterministic Evidence Cluster IDs to eligible solution sections under section 5, set `Evidence Cluster Inventory = Not applicable` for excluded non-solution sections, then freeze the `Section Eligibility Ledger`.

The mapping pass in sections 7 to 19 must consume the frozen ledger row-by-row. Do not alter eligibility or Evidence Cluster IDs after the ledger is frozen.

If a section inventory, eligibility or evidence-cluster error is discovered during mapping, stop the mapping pass and perform at most one full ledger rebuild from Phase 1. Re-run the deterministic eligibility gate, assign eligible-section Evidence Cluster IDs, freeze the corrected ledger, and restart mapping from the corrected frozen ledger. Do not patch the ledger mid-mapping.

If the same ledger issue remains after one rebuild, stop with this deterministic failure message:

`Section Eligibility Ledger reconciliation could not be completed deterministically. Review the source document structure and run the task again.`

Do not perform suggested-control matching for `Excluded non-solution section` sections.

Do not output `Excluded non-solution section` sections in the final action register.

Do not mark excluded non-solution sections as `No ISM control identified`.

Explicit valid `ISM-####` references found in excluded non-solution sections must never be mapped from the excluded section.

If the same control also appears in an eligible solution section, map only the eligible section-control pair. Still count the excluded section-control pair separately as `Valid explicit ISM references found in excluded sections and not mapped`.

Malformed or typo-like ISM references found in excluded non-solution sections may be included only in the separate malformed-reference report, not in the final action register.

Internally record excluded sections and exclusion reasons in the canonical `Section Eligibility Ledger`.

Output only a short exclusion summary before the final output:

`Excluded non-solution sections: <count>.`

Do not include the excluded section list in chat unless requested.

## 9. Existing referenced control quality and relevance check

For every explicit valid `ISM-####` reference found in an eligible numbered `Doco` solution section, do not assume that the referenced control is a good match merely because it appears in the document.

Each explicitly referenced control must be preserved in the ISM Action Register for traceability and assessed against the actual non-comment content assigned to that exact section in the frozen `Section Eligibility Ledger`.

This is a quality and relevance check of controls already referenced in `Doco`. The purpose is to identify whether the document's existing reference agrees with the implementation evidence in that section and, where it does not, whether a better fitting control from the selected ASD OSCAL ISM catalog should be suggested as an alternate.

Do not remove, hide or replace explicitly referenced controls. Always preserve explicitly referenced controls from eligible solution sections in the output, including controls that receive Evidence Score `0`, `1` or `2`, and controls not present in the selected catalog after mandatory validation.

For every explicitly referenced eligible section-control pair:

1. tie the explicit reference to exactly one Evidence Cluster ID from the frozen `Section Eligibility Ledger`;
2. record an `Existing explicit reference` row in the `Candidate Assessment Ledger` for auditability;
3. assign Evidence Score using only the canonical Evidence Score definitions in section 14;
4. compare the section evidence against the exact selected-catalog OSCAL control statement first; and
5. convert the score to the existing-control finding using the table below.

Section 14 is the single source of truth for Evidence Score definitions. Do not use a separate existing-control scoring model.

For each explicitly referenced control, assign one of the following internal `Referenced Control Relevance Assessment` values:

* `Agrees with section evidence`;
* `Partially agrees with section evidence`;
* `Does not agree with section evidence`;
* `Insufficient evidence to assess`;
* `Not present in selected ASD ISM catalog`.

Use these score-to-finding mappings:

| Evidence Score | Referenced Control Relevance Assessment | Existing Control Finding        | Action to Take                     |
| -------------- | --------------------------------------- | ------------------------------- | ---------------------------------- |
| `3`     | Agrees with section evidence            | No issue identified             | Keep existing control              |
| `2`            | Partially agrees with section evidence  | Minor relevance concern         | Review existing control            |
| `0` or `1`     | Does not agree with section evidence    | Potentially incorrect reference | Review existing control            |
| `0` or `1`     | Insufficient evidence to assess         | Insufficient evidence           | Review existing control            |
| n/a            | Not present in selected ASD ISM catalog | Catalog validation issue        | Catalog validation review required |

Use these meanings only for the relevance assessment:

* `Agrees with section evidence` means Evidence Score `3`: the section evidence directly or substantively supports the exact ISM control statement, without relying on broad topic matching or assumptions.
* `Partially agrees with section evidence` means Evidence Score `2`: the referenced control is related to the same Evidence Cluster ID, but that evidence only partly supports the substantive requirement of the referenced control or more directly supports another control present in the selected catalog.
* `Does not agree with section evidence` means Evidence Score `0` or `1` and the local evidence clearly points away from the referenced control or clearly supports a different control objective. Use this only where the mismatch is clear from the section content and the exact ISM control statement.
* `Insufficient evidence to assess` means Evidence Score `0` or `1` and the section explicitly references the control but does not contain enough implementation-specific evidence to confirm whether the control is appropriate.
* `Not present in selected ASD ISM catalog` means the explicit reference was found in `Doco`, but the normalised control ID is not present in the selected ASD OSCAL ISM catalog after section 10 validation.

Make `Insufficient evidence to assess` hard to use. Do not mark `Insufficient evidence to assess` where the same Evidence Cluster ID contains a specific implementation mechanism, configuration, system, process, responsibility, data flow, log source, access model, backup item, vulnerability activity or security tool function that partially supports the referenced control. Use `Partially agrees with section evidence` and `Minor relevance concern` instead.

Where an explicit ISM reference appears in a dedicated control-mapping, compliance-mapping or traceability table, treat the same table row as the primary Evidence Cluster ID. Use only the evidence in that row and any directly referenced eligible numbered section, requirement row, design row or implementation note. Do not use unrelated rows in the table as evidence for that control.

Do not assess agreement using only:

* ISM topic names;
* ISM guideline names;
* broad keywords;
* general cyber security relevance;
* assumptions about what the design probably intended;
* prior knowledge of ISM control families;
* replacement, successor or related controls.

Do not mark a referenced control as wrong merely because another control might also apply.

Only mark a referenced control as `Does not agree with section evidence` where the mismatch is clear from the section content and the exact ISM control statement.

When uncertain between `Does not agree with section evidence` and `Insufficient evidence to assess`, prefer `Insufficient evidence to assess`.

When uncertain between `Insufficient evidence to assess` and `Partially agrees with section evidence`, prefer `Partially agrees with section evidence` if there is any local implementation-specific evidence that partially supports the control statement.

## 12. Suggested alternate controls for existing referenced controls

For every explicit ISM reference in an eligible solution section, perform an active better-fit check against the same local Evidence Cluster ID recorded in the frozen `Section Eligibility Ledger` and resolved through the frozen `Evidence Cluster Ledger`. Record the assessment in the `Candidate Assessment Ledger` whether an alternate is accepted or rejected.

A `Suggested alternate control` is a quality-review finding for an existing explicit reference. It is not a general recommendation, complementary control, related control, successor/replacement control or broader topic suggestion.

### 12A. Mandatory suggested-alternate review decision

For every `Existing control in doc` row, the `Candidate Assessment Ledger` must contain exactly one suggested-alternate review decision for the same `Ledger Row ID`, `Evidence Cluster ID` and `Existing Control`. That review decision must record `Basis Existing Register Row ID` and `Basis Existing Control` so the alternate is traceable to the exact explicit-control row being reviewed.

For every existing-control row with finding `No issue identified`, the required review decision is normally:

`Rejected suggested-alternate review â€” no stronger same-evidence alternate identified`

For every existing-control row with finding `Minor relevance concern`, `Potentially incorrect reference` or `Insufficient evidence`, the review decision must be one of:

* `Accepted suggested alternate`; or
* `Rejected suggested-alternate review â€” no stronger same-evidence alternate identified after named candidate review`; or
* `Rejected suggested-alternate review â€” no same-evidence alternate candidate generated`.

Do not use the shorter rejection reason `No stronger same-evidence alternate identified` for weak existing controls unless the row also satisfies the candidate-discovery detail requirements in section 12C. For weak existing controls, a blank `Candidate ISM Control` is allowed only where `Alternate Candidates Considered = 0` and `Alternate Discovery Method` records the deterministic candidate-discovery method attempted.

If the suggested-alternate review decision is missing for any eligible existing-control row, the Repeatability Audit must fail and `Intra-run validation checks passed` must be `No`.

### 12B. Mandatory same-evidence alternate candidate discovery for weak existing controls

For each existing-control row whose finding is `Minor relevance concern`, `Potentially incorrect reference` or `Insufficient evidence`, generate a same-evidence alternate candidate pool before recording the suggested-alternate review decision.

Use only the same Evidence Cluster ID as the existing-control row. Do not use another section, another evidence cluster, a parent section, a child section, a related section, prior knowledge or a broad topic assumption to create an alternate.

Generate same-evidence alternate candidates using these deterministic passes in this order:

1. `Same-evidence statement-term pass` â€” compare the Evidence Cluster text's required action terms, subject/component terms and security outcome/scope terms against the selected catalog control statements.
2. `Same-evidence topic-neighbour pass` â€” inspect controls in the same catalog topic or immediate neighbouring catalog topic only where at least one statement term from the Evidence Cluster text also appears in the candidate control statement.
3. `Same-evidence explicit-control-neighbour pass` â€” inspect controls near the explicit existing control in catalog order only where the candidate still shares a required action, subject/component or outcome/scope term with the Evidence Cluster text.
4. `Same-evidence locked-intent pass` â€” use one or two locked local intent labels from section 13 only where the label is directly expressed in the Evidence Cluster text.

Deduplicate candidates by normalised ISM control ID. Exclude the existing referenced control itself. Sort the same-evidence alternate candidate pool before scoring by:

1. exact required-action term match count descending;
2. exact subject/component term match count descending;
3. exact security-outcome/scope term match count descending;
4. catalog `sort-id`; and
5. normalised ISM control ID ascending.

Assess up to five strongest same-evidence alternate candidates for each weak existing-control row. If more than five candidate controls are found, reject the additional candidates after the first five with `Candidate Rejection Reason = Stronger candidate selected` and include them in the candidate pool hash only if they were within the deterministic pre-scoring candidate pool.

For each assessed same-evidence alternate candidate, create a Candidate Assessment Ledger row with `Candidate Source Pass = Suggested-alternate better-fit check`, the candidate control ID populated, the same `Basis Existing Register Row ID`, the same `Basis Existing Control`, the same `Ledger Row ID` and the same `Evidence Cluster ID` as the existing-control row.

### 12C. Required audit detail for rejected suggested-alternate reviews

Every rejected suggested-alternate review row for a weak existing control must populate these Candidate Assessment Ledger fields:

* `Alternate Discovery Method` = one or more of `Same-evidence statement-term pass`, `Same-evidence topic-neighbour pass`, `Same-evidence explicit-control-neighbour pass`, `Same-evidence locked-intent pass`, separated by `; ` in the order attempted;
* `Alternate Candidates Considered` = integer count of same-evidence alternate candidate controls assessed for the existing-control row;
* `Best Rejected Alternate Control` = the highest-ranked rejected alternate candidate control ID, or blank only where `Alternate Candidates Considered = 0`;
* `Best Rejected Alternate Failure Reason` = one locked failure reason explaining why the best rejected alternate was not accepted, or `No same-evidence alternate candidate generated` where the count is zero.

Use only these `Best Rejected Alternate Failure Reason` values:

* `Not applicable`;
* `No same-evidence alternate candidate generated`;
* `Existing control remained strongest statement-level match`;
* `Alternate required action not stated`;
* `Alternate subject/component not stated`;
* `Alternate security outcome/scope not stated`;
* `Alternate statement directness below action-register threshold`;
* `Alternate depended on product capability assumption`;
* `Alternate introduced foreign subject`;
* `Alternate was complementary but not better fit`;
* `Alternate tied with existing control and tie could not be resolved`;
* `Alternate rejected by section or evidence-cluster cap`.

For weak existing controls, the rejection reason `No stronger same-evidence alternate identified` is not sufficient by itself. It is valid only when accompanied by the four audit fields above. If those fields are blank or inconsistent, fail RAC-033B and set `Intra-run validation checks passed = No`.

### 12D. Suggested alternate acceptance gate

A `Suggested alternate control` is allowed only when all of the following are true:

1. the section contains an explicit existing `ISM-####` reference;
2. the existing control row has one of these findings:
   * `Minor relevance concern`;
   * `Potentially incorrect reference`;
   * `Insufficient evidence`;
3. the alternate candidate is supported by the same local Evidence Cluster ID as the explicit reference;
4. the alternate candidate receives Evidence Score `3`;
5. the alternate candidate is a stronger statement-level match than the existing referenced control for that same evidence cluster;
6. the alternate candidate does not introduce a control subject, technology, account type, system, data type, process, role or obligation absent from the Evidence Cluster text or immediately adjacent local wording in the same frozen ledger row;
7. the rationale states why the alternate is stronger than the named existing control.

Do not create a `Suggested alternate control` where the existing referenced control has Evidence Score `3` or finding `No issue identified`, unless the alternate receives Evidence Score `3`, the existing control receives Evidence Score `2` or lower, and the alternate is demonstrably a stronger statement-level match for the exact same evidence cluster.

Treat evidence as the same evidence cluster only where the existing ISM reference and the implementation evidence are part of the same sentence, bullet, table row, paragraph, bullet group, table row group, implementation statement, immediately adjacent evidence block, or directly referenced eligible section.

Where an explicit ISM reference appears in a dedicated control-mapping, compliance-mapping or traceability table, treat the same table row as the primary evidence block. Also use any directly referenced eligible numbered section, requirement row, design row or implementation note that the table row points to. Do not use unrelated rows in the table as evidence for that control.

## 13. Suggested controls â€” balanced evidence-led matches

1. candidate whose required action appears verbatim or near-verbatim in the evidence block;
2. candidate whose control subject appears verbatim or near-verbatim in the evidence block;
3. candidate whose security outcome appears verbatim or near-verbatim in the evidence block;
4. candidate with the more specific control statement;
5. candidate with the lower normalised ISM control ID.

If still tied, reject all tied candidates from the ISM Action Register and place only the lower normalised ISM control ID in Optional Review Candidates if Evidence Score `2` and Optional Review Candidates are enabled.

For Evidence Score `3` ties where the rule above still cannot identify one clear candidate, reject all tied candidates rather than including multiple controls for the same evidence cluster. Record `Stronger candidate selected` or `Duplicate evidence cluster already represented`, as applicable, in the `Candidate Assessment Ledger`.

## 14. Action-register evidence-led matching gates

Before including any `Suggested alternate control` or `Suggested control` in the ISM Action Register, apply the following gates to the evidence assigned to the relevant Evidence Cluster ID in the frozen `Section Eligibility Ledger` and recorded in the `Candidate Assessment Ledger`.

The final action-register decision must be copied from an accepted `Candidate Assessment Ledger` row. Do not promote a candidate directly from prose review, memory, catalog browsing or unrecorded reasoning.

### Canonical Evidence Score model

Section 14 is the single source of truth for Evidence Score definitions. Other sections may map Evidence Scores to findings or outputs, but must not redefine the score meanings.

Assign Evidence Score before deciding whether a candidate can enter the ISM Action Register, Optional Review Candidates, or be omitted.

Do not use Evidence Score `4`. Direct and substantive statement-level matches are both scored as `3` to remove score-threshold drift between runs.

| Evidence Score | Label | Permitted treatment |
| -------------- | ----- | ------------------- |
| `3` | High-confidence statement-level match | May be included in the ISM Action Register if all gates pass. |
| `2` | Plausible section-specific near match | Must not be included in the ISM Action Register. May be included only as an Optional Review Candidate when section 13A permits it. |
| `1` | Topic, product-name or keyword-only association | Omit from suggested-control, suggested-alternate and optional-candidate outputs. |
| `0` | No local evidence relationship | Omit from suggested-control, suggested-alternate and optional-candidate outputs. |

Evidence Score `3` = High-confidence statement-level match. All of the following must be true:

* the evidence is an exact quote or faithful extraction from the same eligible section;
* the evidence clearly identifies the control subject/component;
* the evidence clearly identifies the required action, configuration, process or enforcement behaviour;
* one security outcome, condition, scope, constraint or target is explicit in the same Evidence Cluster or directly stated by adjacent local wording in the same frozen ledger row; and
* the match can be explained against the exact catalog control statement without relying on product capability, catalog topic, common practice, evidence from another section, or assumptions outside the frozen ledger row.

Evidence Score `2` = Plausible section-specific near match. Use score `2` where the candidate is section-specific and implementation-evidence-based but one or more Score `3` requirements is not met, including where:

* the evidence supports the general topic but not the exact required action;
* the evidence identifies a component but not the required control behaviour;
* the evidence identifies a security activity but not the control subject or scope;
* the evidence is implementation-specific but only partially satisfies the control statement;
* the match requires an inference beyond adjacent local wording in the same frozen ledger row; or
* the match is plausible enough for human review but not strong enough for action-register inclusion.

Evidence Score `1` = Weak or generic association. Use score `1` where the match is based on broad keywords, product names, headings, capability names, framework references, generic design intent, generic operational benefit, or likely but unstated product capability.

Evidence Score `0` = No defensible relationship. Use score `0` where there is no section-local evidence for the candidate control.

### Structured score-field cap

The `Candidate Assessment Ledger` structured scoring fields are mandatory controls on the Evidence Score. Do not assign Evidence Score `3` unless `Required Action Evidence` and `Subject / Component Evidence` are populated from the same Evidence Cluster ID. Evidence Score `3` requires `Security Outcome / Scope Evidence` from the same Evidence Cluster ID. Evidence Score `3` may use adjacent local wording in the same frozen ledger row for `Security Outcome / Scope Evidence`, but that wording must be quoted or faithfully extracted.

If `Required Action Evidence`, `Subject / Component Evidence`, `Security Outcome / Scope Evidence`, `Assumption Check` or `Score Justification` is blank for an assessed candidate, the maximum permitted Evidence Score is `2`. If `Assumption Check` records `Product capability assumed`, `Topic-only association`, `Foreign subject introduced` or `Evidence outside frozen ledger row`, the maximum permitted Evidence Score is `1`, except that `Foreign subject introduced` must usually be rejected unless a separate local evidence cluster directly states the foreign subject.

### Foreign-subject cap

If the candidate ISM control statement contains a control subject, account type, technology, system, data type, environment, process, role or obligation that is not present in the Evidence Cluster text and is not directly stated in adjacent local wording in the same frozen ledger row, the candidate must not receive Evidence Score `3`.

Where the foreign subject is only generally related to the section topic, assign Evidence Score `1` and use `Candidate Rejection Reason = Foreign subject introduced`. Where the section-specific evidence is plausible but incomplete and does not introduce a materially different subject, assign Evidence Score `2` and consider Optional Review Candidate only.

`Evidence Score = Not applicable` may be used only in the `Candidate Assessment Ledger` for no-candidate support records under section 13B. `Evidence Score = Not applicable â€” control not present in selected catalog` may be used only for explicit existing-control rows where the control is not present in the selected catalog after section 10 validation. Neither value may appear in Optional Review Candidates.

For `Existing control in doc` rows, preserve the explicit reference in the ISM Action Register even if the Evidence Score is `0`, `1` or `2`, but the finding must be one of the allowed existing-control findings under section 9.

A candidate may be included in the ISM Action Register as `Suggested control` or `Suggested alternate control` only where the Evidence Score is `3` and every applicable gate below is satisfied.

Do not use Evidence Score `2` candidates in the ISM Action Register even when the section has no other mapped control. Use Optional Review Candidates when section 13A permits it.

### Gate 1 â€” Catalog presence gate

The control must be present as an `ISM-control` object in the selected ASD OSCAL ISM catalog.

Match by the candidate's normalised `ISM-####` ID against the complete internal catalog index.

Do not suggest controls that are not in the selected catalog.

Do not suggest replacement, successor, superseded, related, external, non-catalog or inferred controls.

### Gate 2 â€” Eligible section gate

The source section must be an `Eligible solution section` in the frozen `Section Eligibility Ledger`.

Do not create suggested-control, suggested-alternate or optional-candidate rows for excluded sections.

### Gate 3 â€” Statement relationship gate

The evidence must support the substantive requirement of the exact ISM control statement, not merely the control intent, control topic, subsection or guideline area.

Control intent may be used only to discover candidates. It must not be used as the final reason for including a `Suggested control` or `Suggested alternate control` in the ISM Action Register.

The match can be included where the section evidence substantively satisfies, supports, enforces, implements or is governed by the ISM control statement, even if the section does not restate every element of the control wording.

If the ISM control statement cannot be connected to the section evidence without relying on broad topic association or unstated assumptions, do not include the control in the ISM Action Register. Consider Optional Review Candidates only if the candidate receives Evidence Score `2` and section 13A permits it.

### Gate 3A â€” Suggested-control precision gate

For `Suggested control` rows, the evidence must match the candidate ISM control statement more precisely than a broad control-intent match.

The rationale must be able to show:

* what the section explicitly says;
* which specific requirement of the ISM control statement is supported;
* which component, process, account type, system, data flow, security tool, configuration or operational activity the control applies to; and
* why the match is stronger than a general topic-level association.

Do not include a `Suggested control` where the rationale would need to rely on wording such as:

* the section is about the same topic;
* the technology normally provides this capability;
* this would usually be expected;
* the control may be relevant;
* the implementation probably includes this;
* this is a good-practice addition;
* the control belongs to the same ISM topic or subsection.

If this gate is only partially met, use Optional Review Candidates rather than the ISM Action Register.

### Gate 4 â€” Specificity gate

The evidence identifies the relevant component, system, account type, network path, log source, protocol, cryptographic setting, backup item, patching process, vulnerability process, security tool, integration, technology role, control responsibility or operational activity.

### Gate 5 â€” No-assumption gate

The match does not require unsupported assumptions about architecture, maturity, effectiveness, governance, ownership, operational practice, scope, inheritance, assurance or controls implemented elsewhere.

Reasonable interpretation of explicit section evidence is allowed. Do not require the document to spell out operational details that are not necessary to establish the control relationship.

### Gate 6 â€” Best-control gate

The control is one of the strongest available statement-level matches for that exact section and Evidence Cluster ID, not one of several broadly related controls.

Where multiple candidate controls appear relevant to the same Evidence Cluster ID, select only the strongest statement-level match using the deterministic tie-break rule in section 13B. Where a candidate is plausible but not one of the strongest statement-level matches, or where two candidates are close but neither is clearly supported at statement level, consider the strongest candidate for Optional Review Candidates rather than the ISM Action Register.

### Gate 7 â€” Lowest-level section gate

The control is mapped to the most specific numbered section containing the evidence, not to a parent/container section unless the parent itself contains the actual evidence.

Do not copy controls from parent sections to child sections.

Do not copy controls from child sections to parent sections.

### Gate 8 â€” Evidence explanation gate

The reason for applicability can be explained using specific wording or evidence from the `Doco` section.

The reason or review finding must not rely on generic phrases such as:

* `broadly relates to`;
* `supports` without explaining how;
* `is relevant to`;
* `may help with`;
* `could be considered`;
* `is good practice for`;
* `typically`;
* `usually`;
* `probably`.

### Gate 9 â€” Candidate ledger gate

The candidate must have a corresponding `Candidate Assessment Ledger` row that records the Evidence Cluster ID, evidence text, candidate control, source pass, Evidence Score, gate result, accepted/rejected status, output destination and applicable rejection reason.

Suggested and alternate action-register rows must be copied only from accepted `Candidate Assessment Ledger` rows with `Output Destination = ISM Action Register`.

Optional Review Candidate rows must be copied only from accepted `Candidate Assessment Ledger` rows with `Output Destination = Optional Review Candidates`.

### Gate failure handling

If a candidate receives Evidence Score `3` but fails any action-register gate, do not promote it into the ISM Action Register. Record the failure in the `Candidate Assessment Ledger` using the locked rejection-reason vocabulary in section 13B.

If a candidate receives Evidence Score `2` and section 13A permits Optional Review Candidates, place it in Optional Review Candidates using the locked `Gate Not Met` vocabulary in section 13A.

If a candidate receives Evidence Score `1` or `0`, omit it from Suggested controls, Suggested alternate controls and Optional Review Candidates. Keep only the Candidate Assessment Ledger record if it was assessed.

## 15. Suggested-control limits and repeatable ranking

For each `Eligible solution section`:

* Prefer one strong Suggested control where the section contains clear implementation-specific security evidence.
* Include a second Suggested control only where the section contains a separate, distinct implementation detail that maps to a different ISM control statement.
* Do not include more than two Suggested controls for any section.
* Do not include more than one Suggested control for any single evidence cluster.
* Do not include more than two suggested alternate controls for any section.
* Do not include more than one suggested alternate control for any single evidence cluster.
* Do not include more than two Optional Review Candidates for any section.
* Do not use these limits as targets.
* Do not add suggested controls to parent/container sections when the same evidence is mapped to a child section.
* Do not add suggested controls to short sections containing only headings, architecture labels, generic descriptions, lists of components or transitional wording.
* Do not add suggested controls for generic references to `ISM alignment`, `PROTECTED`, `Essential Eight`, `security tooling`, `monitoring`, `hardening`, `backup`, `patching`, `MFA`, `RBAC`, `logging`, `SIEM`, `SOAR`, `SOC`, `vulnerability management`, `application control`, `encryption` or `network security` unless the section states the specific implementation detail, solution role, integration, data flow, control responsibility or security function that directly or substantively supports the candidate ISM control statement at statement level.

When more candidates pass the action-register threshold than the section limit permits, rank and keep only the strongest candidates using this exact order:

1. Evidence Score, using `3` as the only action-register score;
2. candidate whose required action appears verbatim or near-verbatim in the evidence block;
3. candidate whose control subject appears verbatim or near-verbatim in the evidence block;
4. candidate whose security outcome appears verbatim or near-verbatim in the evidence block;
5. specificity of local section evidence;
6. completeness of required action / subject / outcome support;
7. distinctness from other selected controls in the same section;
8. lowest-level section fit;
9. candidate with the more specific control statement;
10. candidate with the lower normalised ISM control ID.

Do not include multiple controls for the same Evidence Cluster ID unless the evidence cluster clearly contains separate implementation obligations, separate security activities or separate control outcomes and those obligations are explicitly identified in the rationale.

When two controls are both plausible for the same Evidence Cluster ID, choose the single strongest statement-level match using the deterministic same-score tie-break rule in section 13B and record the reason in the `Candidate Assessment Ledger`. Do not include both unless the evidence block contains two separate implementation obligations. Where neither control is clearly strongest, use the strongest Optional Review Candidate if section 13A permits it; otherwise omit both.

For adjacent sibling sections under the same parent, do not repeat the same suggested control across multiple sections merely because the sections share a common product, architecture area, logging pattern, access model, backup model, hardening theme or control family. Place the control only on the lowest-level section containing the strongest, most specific section-local evidence.

A section describing a security capability does not automatically justify mapping every ISM control related to that capability.

A section may justify one or more controls where it describes how that capability is implemented, integrated, operated, governed by a responsibility model, or used to perform a specific security function in the solution.

For example:

* a section about logging does not automatically justify all logging controls;
* a section about TLS does not automatically justify all cryptography controls;
* a section about privileged access does not automatically justify all access-management controls;
* a section about patching does not automatically justify all vulnerability-management controls;
* a section about backups does not automatically justify all resilience or recovery controls;
* a section about network restrictions does not automatically justify all gateway, firewall or segmentation controls;
* a section about MFA does not automatically justify all authentication controls;
* a section about RBAC does not automatically justify all authorisation, administration or privileged access controls;
* a section about PROTECTED alignment does not automatically justify classification, cryptography, gateway, hardening or governance controls.

## 19A. Mandatory section-count reconciliation gate

* `Existing control in doc`;
* `Suggested alternate control`;
* `Suggested control`;
* `No ISM control identified`.

If it cannot be satisfied after one ledger rebuild, stop with the deterministic ledger-reconciliation failure message in section 6.

For `Mapped controls only` mode, the eligible-section action-register coverage equation is not required because eligible sections without mapped controls are deliberately omitted from the final action register. * `Eligible solution sections` = count of canonical `Section Eligibility Ledger` rows classified as `Eligible solution section`.
* `Excluded non-solution sections` = count of canonical `Section Eligibility Ledger` rows classified as `Excluded non-solution section`.
* `Section count reconciliation passed` = `Yes` only if `Numbered Doco sections identified = Eligible solution sections + Excluded non-solution sections`.

In `Full coverage strict` and `Full coverage`, this count must equal the `No ISM control identified rows` count. In `Mapped controls only`, report the count even though no-control rows are omitted.
* `Explicit ISM section-control pairs in eligible sections` = count of unique eligible section-control pairs, not raw repeated occurrences.
* `Unique discrete explicit ISM controls in eligible sections` = count of unique valid explicit `ISM-####` controls found in eligible solution sections after normalisation and de-duplication.

### Column rules

#### Register Row ID

Use deterministic IDs in final sorted order: `REG-0001`, `REG-0002`, and so on.

#### Ledger Row ID

#### Evidence Cluster ID

Use the exact Evidence Cluster ID from the frozen `Section Eligibility Ledger` and the corresponding `Candidate Assessment Ledger` row.

Every action-register row must reference exactly one Evidence Cluster ID.

For `No ISM control identified` rows, use the strongest rejected candidate Evidence Cluster ID for that section. If an eligible section has no Evidence Cluster ID, stop and correct the `Section Eligibility Ledger` before producing the action register.

#### Doco Numbered Section / Subsection

Use the eligible numbered section from `Doco`.

Use document order, not lexical order.

For example, `3.10` comes after `3.9`.

#### Section Title / Topic

Use the section heading from `Doco`.

If the heading is unclear, provide a concise topic summary based only on the section content.

#### Output Category

Use only:

* `Existing control in doc`;
* `Suggested alternate control`;
* `Suggested control`;
* `No ISM control identified`.

#### Trigger Phrase

Populate this for every row.

Use one concise evidence phrase, configuration detail, table value or setting that triggered the row. The value must be exact local document evidence copied or faithfully extracted from the same eligible section in the frozen `Section Eligibility Ledger`.

Do not use generated labels as trigger phrases, including labels such as:

* `Centralised logging/SIEM integration`;
* `Credential/secret storage and retrieval protections`;
* `Network access controls`;
* `Authentication / authorisation / JWT validation`;
* `MFA / Conditional Access`;
* `Application control implementation`;
* any other model-generated topic, control family, or inferred intent label.

Generated labels may be used internally for candidate discovery, but they must not appear as evidence and must not justify promotion.

If an exact local trigger phrase cannot be quoted or faithfully extracted from the frozen ledger row, the candidate cannot be promoted to `Suggested control` or `Suggested alternate control`.

#### Existing Control

Populate this only where the row relates to a valid explicit ISM control already referenced in an eligible solution section of `Doco`.

For `Suggested alternate control` rows, populate this with the existing control being reviewed.

Leave blank for `Suggested control` rows.

Leave blank for `No ISM control identified` rows.

#### Existing Control Finding

Populate this for `Existing control in doc` rows.

For `Suggested alternate control` rows, repeat the finding for the existing control being reviewed.

Use only:

* `No issue identified`;
* `Minor relevance concern`;
* `Potentially incorrect reference`;
* `Insufficient evidence`;
* `Catalog validation issue`.

Leave blank for `Suggested control` and `No ISM control identified` rows.

#### Suggested Alternate Control

Populate this only for `Suggested alternate control` rows.

Use the normalised `ISM-####` value.

Leave blank for all other row types.

#### Suggested New Control

Populate this only for `Suggested control` rows.

Use the normalised `ISM-####` value.

Leave blank for all other row types.

#### Evidence Score

Populate using the canonical Evidence Score assigned under section 14 and recorded in the Candidate Assessment Ledger. Section 9 maps Evidence Scores to existing-control findings but does not redefine Evidence Score meanings.

For `No ISM control identified` rows, leave blank unless a rejected candidate is referenced in the rationale, in which case use the highest rejected candidate Evidence Score for that section.

#### Action to Take

Use only the allowed action-register values.

Do not create custom action text.

#### Existing Control Description

Use the description-column population rules in section 18.

#### Suggested Control Description

Use the description-column population rules in section 18.

#### Mapped / Suggested ISM Guideline Section, Mapped / Suggested ISM Guideline Subsection and Mapped / Suggested ISM Topic

Populate from the selected ASD OSCAL ISM catalog for the control being actioned in the row.

For `Existing control in doc` rows, these hierarchy fields apply to the existing referenced control.

For `Suggested alternate control` rows, these hierarchy fields apply to the suggested alternate control, not the existing referenced control.

For `Suggested control` rows, these hierarchy fields apply to the Suggested control.

For controls not present in the selected catalog, leave these fields blank.

Do not invent or infer missing hierarchy values.

When uncertain, use the nearest available catalog ancestor titles and leave ambiguous higher-level fields blank rather than guessing.

#### Rationale

Use concise rationale.

For existing controls with finding `No issue identified`, use:

`Keep existing control because the section explicitly states <evidence>, which directly matches the ISM control statement requiring <control intent>.`

For existing controls with finding `Minor relevance concern`, use:

`Review existing control because the section explicitly states <evidence>, which only partially matches the ISM control statement.`

For existing controls with finding `Potentially incorrect reference`, use:

`Review existing control because the section explicitly states <evidence>, which does not directly match the Existing Control statement.`

For existing controls with finding `Insufficient evidence`, use:

`Review existing control because it is explicitly referenced in Doco, but the section does not provide enough implementation-specific evidence to confirm direct applicability.`

For existing controls with finding `Catalog validation issue`, use:

`Catalog validation review required because the control is explicitly referenced in Doco but is not present in the selected ASD OSCAL ISM catalog after mandatory validation.`

For suggested alternates, use:

`Suggested as an alternate because the section explicitly states <evidence>, which substantively supports the suggested ISM control statement requiring <control intent>. The existing referenced control <ISM-####> received finding <Existing Control Finding> because <reason>.`

For Suggested controls, use:

`Add new control because the section explicitly states <specific local evidence>, which directly supports the ISM control statement requirement for <specific action / mechanism / process / configuration / responsibility>. This is a Suggested control rather than an Optional Review Candidate because <statement-level match reason>.`

For `No ISM control identified` rows, use:

`No ISM control identified because the section does not contain enough implementation-specific evidence to support an action-register ISM control.`
